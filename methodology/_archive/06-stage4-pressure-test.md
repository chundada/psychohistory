# Stage 4：压力测试（Pressure Test）

> **⚠️ 已归档废弃**：本环节从未实际执行（416 测试/系列无人运行），现行 3 步流水线不含此阶段。本文件仅作历史存档，请勿遵循。

> **目标**：通过盲测验证技能的有效性——在不知道技能来源的情况下，测试者能否正确触发/不触发
> **输入**：SKILL.md 文件
> **输出**：测试结果报告 + 是否需要重做 Stage 2

---

## 一、为什么需要压力测试

技能是人造的，而人容易产生"知识诅咒"（Curse of Knowledge）——一旦你知道了技能的内容，就很难想象不知道它的人会怎么使用。

压力测试通过**盲测**打破这个诅咒：让一个不了解技能的测试者（或未来的你自己）根据测试提示判断是否触发该技能。

### 通过标准
- **总体通过率 >= 80%** → 进入 Stage 5
- **通过率 < 80%** → 回到 Stage 2，修改对应的 SKILL.md

---

## 二、测试提示设计

### 提示类型与数量要求

每篇 SKILL.md 需要 8-12 个测试提示，分三类：

| 类别 | 数量 | 含义 | 预期结果 |
|------|------|------|---------|
| **should_trigger** | 3-5 | 应该触发该技能的场景 | 测试者应回答"是" |
| **should_not_trigger** | 2-3 | 不应该触发该技能的场景 | 测试者应回答"否" |
| **edge_case** | 1-3 | 边界场景，模糊不清 | 测试者可能回答"不确定" |

### 设计原则

#### should_trigger 原则
- 包含触发信号词（来自 A2 字段）
- 场景具体、接近真实使用场景
- 不直接引用技能名称或术语

**示例**（对于上述"文明脆弱性分析"技能）：
```json
{
  "prompt": "我正在分析一个非洲国家，该国最近 10 年经济快速增长，但出现了严重的部落间冲突。首都从 500 万人口增长到 1200 万，政府在基础设施上投入很大但不同部落获得的资源不均。",
  "expected": "should_trigger"
}
```

#### should_not_trigger 原则
- 场景看起来类似但实际上不适用
- 故意放入技能 B 中列出的不适用场景

**示例**：
```json
{
  "prompt": "我在分析一个 8 人的创业团队，大家都是来自同一所大学的同学，共享相同的价值观和目标。团队最近因为分工问题有一些小分歧。",
  "expected": "should_not_trigger"
}
```

#### edge_case 原则
- 测试技能的边界
- 故意模糊触发条件

**示例**：
```json
{
  "prompt": "我在看一个城市的发展规划，该城市有 6 个主要的利益群体（房地产商、环保组织、中小企业工会、教育机构、本地政党和移民社区），市政府建立了一个多方协商平台，但这个平台刚运行 3 个月。",
  "expected": "edge_case"
}
```

## 三、测试流程

### 准备阶段

1. **编写 test-prompts.json**：为技能创建测试提示文件
2. **盲测设置**：确保测试者在测试前没有阅读该 SKILL.md

### 执行阶段

3. **展示技能信息**：只展示技能的名称和 A2（触发条件）部分
4. **逐一展示提示**：每次展示一个提示，测试者回答"会触发/不会触发/不确定"
5. **记录结果**：记录每个提示的测试结果

### 评估阶段

6. **计算通过率**：

```
通过率 = (正确触发的 should_trigger 数 + 正确拒绝的 should_not_trigger 数) / (总 should_trigger + 总 should_not_trigger)
```

注意：edge_case 不参与通过率计算，只作为参考。

7. **判断结果**：
   - >= 80% → 通过，进入 Stage 5
   - < 80% → 不通过，分析失败模式，返回 Stage 2

## 四、测试结果报告格式

```markdown
# 压力测试报告：<技能名称>

**测试日期**：YYYY-MM-DD
**测试员**：（如果他人测试，记录姓名）

## 结果汇总

| 类型 | 总数 | 正确 | 错误 | 通过率 |
|------|------|------|------|--------|
| should_trigger | 3 | 3 | 0 | 100% |
| should_not_trigger | 2 | 1 | 1 | 50% |
| edge_case | 2 | - | - | （参考） |
| **总体** | 5 | 4 | 1 | **80%** ✅ |

## 失败分析

### 失败项 #1
- **提示**：<提示内容>
- **预期**：should_not_trigger
- **实际**：should_trigger
- **分析**：A2 的触发条件覆盖了不该覆盖的场景。
  B（边界条件）中的"小规模同质群体"写得不够明显。
- **改进建议**：在 A2 中增加"群体规模超过 20 人"的条件。

## 改进计划

1. 修改 A2 触发条件，明确最低群体规模
2. 在 B 中增加"8 人以下且高度同质的群体"作为明确的不适用场景
3. 重新测试
```

## 五、test-prompts.json 模板使用

使用 `test-prompts.json.template`（与本文件同目录，已一并归档）：

```json
{
  "skill": "GT-civilization-fragility-analysis",
  "test_suite": {
    "should_trigger": [
      {
        "id": "T01",
        "prompt": "...",
        "expected": "should_trigger",
        "rationale": "为什么这个应该触发"
      }
    ],
    "should_not_trigger": [
      {
        "id": "N01",
        "prompt": "...",
        "expected": "should_not_trigger",
        "rationale": "为什么这个不应该触发"
      }
    ],
    "edge_case": [
      {
        "id": "E01",
        "prompt": "...",
        "expected": "edge_case",
        "rationale": "为什么这个是边界情况"
      }
    ]
  }
}
```

## 六、失败模式与对应修改

| 失败模式 | 问题根源 | 修改位置 |
|---------|---------|---------|
| should_trigger 被 miss | A2 触发条件写得太窄 | 扩大 A2 的范围 |
| should_not_trigger 被误触发 | A2 触发条件写得太宽 | 缩小 A2 的范围，或在 B 中增加排除条件 |
| should_trigger 触发但用了错误方式 | E 执行步骤不够清晰 | 重写 E 步骤 |
| edge_case 结果不理想 | 技能的边界定义不清晰 | 在 B 中增加边界说明 |

## 八、常见陷阱

| 陷阱 | 后果 | 避免方法 |
|------|------|---------|
| 自我测试（自己写自己测） | 结果失真，失去盲测意义 | 让另一个 AI 或人测试 |
| should_trigger 写得太明显（直接引用技能术语） | 测试结果无效 | 使用自然语言描述场景，不用技能术语 |
| should_not_trigger 设计太少 | 无法充分测试"拒真"能力 | 至少 2 个 |
| 忽略 edge_case | 边界模糊时用户不知如何选择 | 至少 1 个 |
| 通过即止，不优化 | 技能质量停留在"及格线" | 通过后继续优化到 90%+ |
